# Feature Specification: 설정 별도 창과 단축어 실행

**Feature Branch**: `017-settings-window-shortcut`

**Created**: 2026-07-08

**Status**: Draft

**Input**: User description: ""설정" 화면을 별도 window 로 분리하고 cmd + , 단축어로 실행할 수 있도록 한다"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 설정을 별도 창에서 열기 (Priority: P1)

사용자는 현재 작업 화면을 유지한 채 설정 화면을 별도 창으로 연다. 설정을 확인하거나 변경하는 동안 기존 작업 세션, 패널, 입력 상태를 잃지 않고 다시 작업 화면으로 돌아올 수 있다.

**Why this priority**: 기능의 핵심 가치다. 설정 화면이 작업 화면 안에 섞이면 사용자가 진행 중인 세션 맥락을 잃거나 화면 이동 비용을 치르게 된다.

**Independent Test**: 앱에서 설정 열기 동작을 수행했을 때 설정 전용 창이 표시되고, 기존 메인 작업 창의 상태가 그대로 유지되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 메인 작업 창에서 세션을 보고 있는 상태, **When** 사용자가 설정을 열면, **Then** 설정 전용 창이 별도로 표시되고 메인 작업 창은 닫히거나 다른 화면으로 전환되지 않는다.
2. **Given** 설정 창이 열린 상태, **When** 사용자가 메인 작업 창으로 돌아가면, **Then** 설정 창을 닫지 않아도 기존 작업 화면을 계속 사용할 수 있다.
3. **Given** 설정 창에서 값을 변경한 상태, **When** 변경 사항이 저장되면, **Then** 설정이 필요한 앱 동작에 반영되고 사용자는 메인 작업 흐름을 다시 시작하지 않아도 된다.

---

### User Story 2 - `Cmd+,` 단축어로 설정 열기 (Priority: P2)

사용자는 macOS에서 익숙한 설정 단축어인 `Cmd+,`를 눌러 어디서든 설정 창을 빠르게 연다.

**Why this priority**: 설정 접근성을 높이고 macOS 앱 관례에 맞춘다. 메뉴나 화면 내 버튼을 찾지 않아도 반복적으로 설정을 열 수 있다.

**Independent Test**: 메인 작업 창이 활성화된 상태에서 `Cmd+,`를 누르면 설정 창이 열리거나 이미 열린 설정 창이 앞으로 오는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 메인 작업 창이 활성화된 상태, **When** 사용자가 `Cmd+,`를 누르면, **Then** 설정 창이 열린다.
2. **Given** 설정 창이 이미 열린 상태, **When** 사용자가 `Cmd+,`를 다시 누르면, **Then** 새 설정 창이 중복 생성되지 않고 기존 설정 창이 사용자에게 표시된다.
3. **Given** 사용자가 텍스트 입력 중인 상태, **When** `Cmd+,`를 누르면, **Then** 입력 중인 텍스트에 쉼표가 삽입되는 대신 설정 열기 동작이 실행된다.

---

### User Story 3 - 기존 설정 진입점의 일관성 유지 (Priority: P3)

사용자는 기존 메뉴나 화면 내 설정 진입점을 사용해도 동일한 설정 창을 연다. 설정 접근 방식이 여러 개여도 결과와 상태는 일관적이어야 한다.

**Why this priority**: 기존 사용자의 습관과 발견 가능성을 보존한다. 단축어만 추가하면 설정을 찾는 신규 사용자의 경험이 약해질 수 있다.

**Independent Test**: 기존 설정 버튼 또는 메뉴 항목으로 설정을 열고, 단축어로 연 경우와 동일한 창 동작 및 중복 방지 규칙이 적용되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 기존 설정 버튼 또는 메뉴 항목이 있는 상태, **When** 사용자가 해당 진입점을 선택하면, **Then** 설정 전용 창이 열린다.
2. **Given** 설정 창이 이미 열린 상태, **When** 사용자가 기존 설정 진입점을 다시 선택하면, **Then** 기존 설정 창이 앞으로 오고 중복 창은 생기지 않는다.
3. **Given** 사용자가 설정 창을 닫은 상태, **When** 단축어 또는 기존 진입점으로 다시 열면, **Then** 마지막으로 저장된 설정 값이 표시된다.

### Edge Cases

- 설정 창이 이미 열려 있을 때 단축어, 메뉴, 버튼을 반복 실행해도 설정 창은 하나만 유지되어야 한다.
- 설정 창이 다른 창 뒤에 가려져 있거나 최소화된 경우 설정 열기 동작은 사용자가 즉시 볼 수 있도록 기존 창을 앞으로 가져와야 한다.
- 메인 창이 여러 작업 상태를 가지고 있어도 설정 창을 열거나 닫는 동작이 현재 세션, 입력 중인 프롬프트, 선택된 패널을 초기화하지 않아야 한다.
- 설정 창을 닫는 동작은 앱 전체 종료나 메인 작업 창 종료로 이어지지 않아야 한다.
- 설정 저장 실패나 유효하지 않은 설정 입력이 발생하면 설정 창 안에서 사용자가 이해할 수 있는 안내를 제공하고, 메인 작업 창은 계속 사용할 수 있어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 사용자는 설정 화면을 메인 작업 창과 분리된 전용 창으로 열 수 있어야 한다.
- **FR-002**: 설정 창을 여는 동작은 메인 작업 창의 현재 작업 상태, 세션 표시, 입력 상태를 초기화하거나 닫지 않아야 한다.
- **FR-003**: 시스템은 `Cmd+,` 단축어를 제공해 활성 창에서 설정 창을 열 수 있어야 한다.
- **FR-004**: 설정 창이 이미 열려 있을 때 설정 열기 동작이 다시 실행되면, 시스템은 새 창을 만들지 않고 기존 설정 창을 사용자에게 표시해야 한다.
- **FR-005**: 기존 설정 진입점이 있다면 해당 진입점은 별도 설정 창 열기 동작과 동일한 결과를 제공해야 한다.
- **FR-006**: 사용자는 설정 창을 닫아도 메인 작업 창과 앱 실행 상태를 유지할 수 있어야 한다.
- **FR-007**: 설정 창에서 저장된 변경 사항은 사용자가 메인 작업 흐름을 다시 시작하지 않아도 이후 관련 동작에 반영되어야 한다.
- **FR-008**: 설정 창에서 저장 실패 또는 입력 오류가 발생하면, 시스템은 설정 창 안에서 문제를 안내하고 사용자가 수정 후 다시 시도할 수 있게 해야 한다.
- **FR-009**: 설정 창은 앱의 기존 설정 항목을 동일한 의미와 기본값으로 표시해야 하며, 별도 창으로 이동하면서 설정 항목이 누락되면 안 된다.

### Key Entities

- **설정 창**: 앱 설정을 표시하고 편집하는 전용 창. 열림/닫힘/앞으로 가져오기 상태를 가진다.
- **설정 진입점**: 사용자가 설정 창을 여는 방법. 단축어, 메뉴 항목, 화면 내 버튼을 포함한다.
- **설정 값**: 앱 동작을 제어하는 사용자 구성 값. 설정 창에서 조회·수정·저장된다.

## Constitution Alignment *(mandatory)*

- **Monorepo boundary**: `apps/agentic-workbench` 단독 범위다. 설정 화면과 창 동작은 Agentic Workbench 앱에 한정하며, cross-app 공유는 요구하지 않는다.
- **Frontend layering**: 설정 화면 조합과 창 전용 화면은 `app`/`pages`, 설정 열기 사용자 동작과 단축어 진입은 `features`, 설정 도메인 모델과 저장 어댑터 호출은 `entities`, 재사용 UI는 필요 시 `shared` 또는 기존 `components/ui` 범위에서 다룬다.
- **Backend boundary**: 별도 창 생성과 설정 저장이 앱 셸 또는 Tauri 경계를 통과해야 하는 경우, inbound 명령은 입력 검증과 위임만 담당하고 설정 저장·조회 규칙은 application service와 repository/port 뒤에 둔다. 순수 설정 모델은 domain에 유지한다.
- **Shared core vs UI**: 이번 기능은 Agentic Workbench 전용 UX이므로 공유 UI 패키지로 승격하지 않는다. 설정 값 검증이나 정규화가 둘 이상의 영역에서 필요해질 때만 headless helper로 분리한다.
- **Persistence and safety**: 설정 저장은 기존 설정 저장소와 동일한 경로·권한 범위를 사용해야 한다. 설정 창 열기/닫기 동작은 실행 중인 세션 소유권, 작업 디렉터리, 권한 상태를 변경하지 않아야 한다.
- **Documentation and Storybook**: 설정 창 화면의 기본/오류/긴 내용 상태 Storybook 등록을 검토한다. 사용자 문서가 설정 접근 방법을 설명하고 있다면 `docs/*.md` 한국어 문서를 갱신한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자는 메인 작업 화면에서 `Cmd+,`를 눌러 1초 이내에 설정 창 표시 또는 기존 설정 창 포커스를 확인할 수 있다.
- **SC-002**: 설정 열기 동작을 5회 연속 실행해도 활성 설정 창은 1개만 유지된다.
- **SC-003**: 설정 창을 열고 닫은 뒤에도 메인 작업 창의 현재 세션, 선택된 패널, 입력 중인 프롬프트가 유지된다.
- **SC-004**: 기존 설정 진입점과 `Cmd+,` 단축어가 모두 동일한 설정 창을 열며, 사용자 테스트에서 두 경로의 결과 차이가 발견되지 않는다.
- **SC-005**: 사용자는 설정 변경 후 메인 작업 흐름을 재시작하지 않고 관련 동작의 변경 반영을 확인할 수 있다.

## Assumptions

- 대상 단축어는 macOS 관례에 맞춰 `Cmd+,`로 정의한다. 다른 운영체제 단축어 대응은 이번 명세의 직접 범위가 아니다.
- 설정 창은 앱 전체에 하나만 존재하는 단일 설정 창으로 동작한다.
- 기존 설정 항목과 저장 의미는 유지하며, 이번 기능은 설정 화면의 위치와 접근 방식을 바꾸는 것이 핵심이다.
- 설정 변경의 즉시 반영 범위는 현재 앱이 이미 지원하는 설정 반영 정책을 따른다. 즉시 반영이 불가능한 항목은 기존과 동일하게 사용자에게 필요한 후속 조치를 안내한다.
