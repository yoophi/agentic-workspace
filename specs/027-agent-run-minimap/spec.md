# Feature Specification: Agent Run 히스토리 미니맵

**Feature Branch**: `027-agent-run-minimap`

**Created**: 2026-07-10

**Status**: Implemented

**Input**: User description: "WorktreeSessionPage의 agent-run 영역을 기존 history와 미니맵으로 좌우 분할하고, 미니맵에서 이전 프롬프트와 응답을 간결하게 확인하며 드래그로 원하는 위치로 스크롤할 수 있게 한다. 미니맵은 보이기/숨기기를 전환할 수 있다."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 긴 대화의 위치를 미니맵으로 탐색 (Priority: P1)

사용자는 긴 agent run을 검토할 때 히스토리 옆의 미니맵에서 사용자 프롬프트와 agent 출력의 전체 흐름을 한눈에 파악하고, 현재 보이는 구간을 원하는 과거 위치로 드래그해 해당 내용을 즉시 확인할 수 있다.

**Why this priority**: 이전 프롬프트와 응답을 찾기 위해 긴 히스토리를 반복 스크롤하는 비용을 줄이는 것이 이 기능의 핵심 가치다.

**Independent Test**: 여러 차례의 사용자 프롬프트와 긴 agent 출력이 포함된 run을 열고, 미니맵의 현재 구간 표시를 과거 프롬프트 위치로 드래그했을 때 해당 프롬프트와 인접 응답이 히스토리 영역에 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 한 화면을 넘는 사용자 프롬프트와 agent 출력이 존재함, **When** 사용자가 agent-run 패널을 봄, **Then** 기존 히스토리 옆 미니맵에 프롬프트와 출력의 축약된 흐름 및 현재 보이는 구간이 구분되어 표시된다.
2. **Given** 미니맵에 현재 보이는 구간이 표시되어 있음, **When** 사용자가 그 구간을 위쪽 과거 항목으로 드래그함, **Then** 히스토리 영역이 드래그 위치에 대응하는 과거 내용으로 이동한다.
3. **Given** 사용자가 미니맵으로 특정 프롬프트 위치로 이동함, **When** 드래그를 끝냄, **Then** 해당 프롬프트와 주변 agent 출력의 관계를 히스토리 영역에서 식별할 수 있다.
4. **Given** 포인터 드래그를 사용할 수 없음, **When** 사용자가 현재 구간 표시에 키보드 초점을 두고 Arrow/Page/Home/End 키를 사용함, **Then** 보조 기술이 조작 목적과 현재 위치를 식별할 수 있고 사용자는 시작, 중간, 끝 위치에 도달할 수 있다.

---

### User Story 2 - 히스토리 스크롤 위치를 미니맵에서 확인 (Priority: P2)

사용자가 기존 히스토리를 마우스 휠, 트랙패드 또는 다른 기존 수단으로 스크롤하면 미니맵의 현재 구간 표시도 함께 이동하여 전체 대화 중 현재 위치를 알려준다.

**Why this priority**: 미니맵 탐색과 기존 스크롤이 서로 다른 위치를 나타내면 탐색 도구로서 신뢰할 수 없으므로 양방향 동기화가 필요하다.

**Independent Test**: 히스토리의 시작, 중간, 끝으로 각각 스크롤하고 미니맵의 현재 구간 표시가 동일한 상대 위치와 범위를 나타내는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 긴 히스토리와 표시 중인 미니맵, **When** 사용자가 히스토리를 시작에서 중간으로 스크롤함, **Then** 미니맵의 현재 구간 표시가 전체 흐름의 중간 위치로 이동한다.
2. **Given** agent가 새 출력을 생성하는 중, **When** 히스토리의 길이가 늘어남, **Then** 미니맵의 축약 내용과 현재 구간 표시가 최신 흐름을 반영하며 사용자가 보고 있던 과거 위치를 임의로 끝으로 이동시키지 않는다.

---

### User Story 3 - 미니맵 표시 공간 제어 (Priority: P3)

사용자는 미니맵이 필요하지 않을 때 숨겨 기존 agent-run 히스토리에 전체 너비를 사용하고, 다시 필요할 때 표시할 수 있다.

**Why this priority**: 미니맵은 탐색을 돕지만 짧은 대화나 좁은 화면에서는 히스토리 가독성을 위해 숨길 수 있어야 한다.

**Independent Test**: 미니맵 토글을 연속으로 사용해 숨김 상태에서는 히스토리가 가용 너비를 차지하고, 다시 표시해도 폭 변경 전후의 논리 viewport 시작 비율이 5% 이내로 유지되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 미니맵이 표시되어 있음, **When** 사용자가 숨기기 토글을 선택함, **Then** 미니맵이 사라지고 히스토리 영역이 해제된 공간을 사용한다.
2. **Given** 미니맵이 숨겨져 있음, **When** 사용자가 보이기 토글을 선택함, **Then** 미니맵이 다시 나타나며 현재 히스토리 위치를 정확히 표시한다.
3. **Given** 사용자가 과거 위치를 보고 있음, **When** 미니맵을 숨겼다가 다시 표시함, **Then** 항목 재배치로 픽셀 `scrollTop`이 바뀌더라도 전체 흐름 대비 논리 viewport 시작 비율은 전환 전 위치와 5% 이내로 유지된다.

### Edge Cases

- 히스토리가 비어 있으면 미니맵은 탐색 가능한 내용이 없음을 나타내며 드래그 동작을 시작하지 않는다.
- 전체 히스토리가 한 화면 안에 들어오면 현재 구간 표시는 전체 미니맵 범위를 나타내고, 드래그해도 불필요한 위치 이동이 발생하지 않는다.
- 매우 긴 프롬프트, 코드 블록, Mermaid 다이어그램, 여러 줄 출력은 미니맵에서 원문 전체를 재현하지 않고 발화 유형과 상대 길이를 식별할 수 있는 축약 형태로 표시한다.
- 도구 호출, 권한 요청, 상태 메시지 등 사용자 프롬프트나 agent 출력이 아닌 이벤트는 미니맵의 핵심 콘텐츠를 방해하지 않아야 한다.
- 스트리밍 출력으로 히스토리 길이가 계속 변하는 동안 과거 내용을 보고 있는 사용자의 위치는 유지되며, 최신 위치를 보고 있던 경우에는 기존 자동 추적 동작이 유지된다.
- run이 전환되거나 추가 agent 패널을 선택하면 미니맵은 활성 패널의 히스토리만 나타내며 다른 패널의 위치나 내용을 섞지 않는다.
- 미니맵을 드래그하는 중 포인터가 미니맵 경계를 벗어나더라도 유효 범위의 시작 또는 끝을 넘겨 스크롤하지 않는다.
- 미니맵이 표시되기 어려울 정도로 가용 너비가 좁아도 히스토리의 핵심 콘텐츠와 조작 요소가 겹치지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 활성 agent-run 패널의 히스토리 영역과 미니맵을 나란히 배치해야 한다.
- **FR-002**: 미니맵은 활성 패널에서 사용자가 입력한 프롬프트와 agent가 생성한 출력을 시간순으로 구분 가능한 축약 형태로 표시해야 한다.
- **FR-003**: 미니맵의 각 축약 항목은 원본의 전체 내용을 복제하지 않으면서 발화 주체, 순서 및 상대적인 콘텐츠 규모를 식별할 수 있어야 한다.
- **FR-004**: 시스템은 전체 히스토리 중 현재 히스토리 영역에 보이는 범위와 위치를 미니맵에 시각적으로 구분해 표시해야 한다.
- **FR-005**: 사용자는 미니맵의 현재 구간 표시를 드래그하여 히스토리의 시작부터 끝 사이 원하는 상대 위치로 이동할 수 있어야 한다.
- **FR-006**: 미니맵 드래그 중 히스토리 영역은 드래그 위치에 연속적으로 대응하여 스크롤되어야 한다.
- **FR-007**: 기존 히스토리 영역의 스크롤 위치 또는 표시 범위가 바뀌면 미니맵의 현재 구간 표시도 같은 위치와 범위를 반영해야 한다.
- **FR-008**: 실행 중 새 프롬프트 또는 agent 출력이 추가되면 미니맵은 시간순 흐름을 갱신해야 한다.
- **FR-009**: 미니맵 갱신은 사용자가 과거 내용을 보고 있는 상태를 임의로 최신 위치로 이동시키지 않아야 하며, 기존의 최신 출력 자동 추적 동작을 방해하지 않아야 한다.
- **FR-010**: 사용자는 명확히 식별 가능한 토글로 미니맵을 보이거나 숨길 수 있어야 한다.
- **FR-011**: 미니맵을 숨기면 히스토리 영역은 미니맵이 차지하던 가용 공간을 사용해야 한다.
- **FR-012**: 미니맵 표시 상태를 전환해도 활성 패널의 전체 흐름 대비 논리 viewport 시작 비율은 전환 전후 5% 이내로 유지되어야 한다. 폭 변경에 따른 항목 재배치로 픽셀 `scrollTop`이 달라지는 것은 허용한다.
- **FR-013**: 여러 agent-run 패널이 존재할 때 미니맵 콘텐츠와 스크롤 상태는 각 패널별 히스토리에 대응해야 하며 활성 패널 간에 섞이지 않아야 한다.
- **FR-014**: 사용자는 포인터 드래그를 사용할 수 없는 경우에도 현재 구간 표시를 키보드로 이동하여 히스토리를 탐색할 수 있어야 한다.
- **FR-015**: 미니맵 토글과 현재 구간 조작 요소는 보조 기술이 목적과 현재 상태를 식별할 수 있는 이름 및 상태 정보를 제공해야 한다.
- **FR-016**: 미니맵은 빈 히스토리, 한 화면 이하의 짧은 히스토리, 스트리밍 중인 히스토리에서 각각 안정적인 비활성 또는 동기화 상태를 제공해야 한다.

### Key Entities

- **Agent Run History**: 활성 agent-run 패널에 시간순으로 표시되는 사용자 프롬프트, agent 출력 및 기타 실행 이벤트의 전체 흐름.
- **Minimap Entry**: 사용자 프롬프트 또는 agent 출력을 발화 주체, 순서, 상대 크기로 축약해 나타내며 원본 히스토리 위치와 대응하는 항목.
- **Viewport Indicator**: 전체 히스토리에서 현재 보이는 시작 위치와 범위를 나타내며 드래그 또는 키보드 조작의 대상이 되는 표시.
- **Minimap Visibility**: 특정 agent-run 패널에서 미니맵이 현재 표시되는지 숨겨졌는지를 나타내는 화면 상태.

## Constitution Alignment *(mandatory)*

- **Monorepo boundary**: 범위는 `apps/agentic-workbench`의 WorktreeSessionPage 내 agent-run 경험에 한정한다. 다른 앱이 동일한 탐색 UI를 소비한다는 요구가 없으므로 cross-app package 또는 crate 공유는 필요하지 않다.
- **Frontend layering**: 화면 배치는 `pages/project-worktree-session`의 기존 조합을 유지하고, 미니맵 탐색과 토글 같은 사용자 상호작용은 `features/agent-run`, 히스토리 항목의 도메인 정보와 축약에 필요한 순수 모델은 `entities/agent-run` 책임으로 구분한다. 범용 조작 요소가 필요할 때만 `shared` 또는 기존 `components/ui`를 사용한다.
- **Backend boundary**: 기존 agent-run 히스토리 데이터를 화면에서 탐색하는 기능이므로 Tauri/Rust backend, application service, persistence adapter 변경은 요구하지 않는다.
- **Shared core vs UI**: app-specific UI로 유지한다. 축약 및 위치 계산처럼 독립 검증 가능한 순수 로직은 UI와 분리하되, 실제로 두 앱 이상에서 요구되기 전에는 workspace package로 추출하지 않는다.
- **Persistence and safety**: 새로운 파일 접근, 권한, agent session 변경 또는 영속 데이터는 없다. 각 패널의 기존 run/session 범위를 준수하고 다른 panel/session의 히스토리를 노출하지 않아야 한다.
- **Documentation and Storybook**: 재사용 가능한 미니맵 컴포넌트에는 atoms/molecules/organisms 분류에 맞는 Storybook 스토리를 등록하고 빈 상태, 짧은 히스토리, 긴 히스토리, 스트리밍 상태, 숨김 상태를 포함한다. 별도 프로젝트 문서는 이번 사용자 기능 명세 범위에서 요구하지 않는다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 20회 이상의 프롬프트-응답 흐름을 사용하는 브라우저 검증에서 모든 사용자 프롬프트가 미니맵의 구분 가능한 항목으로 투영되고, 지정한 시작·중간·끝 프롬프트 위치에 한 번의 연속 드래그 또는 키보드 조작으로 도달한다.
- **SC-002**: 미니맵의 시작, 중간, 끝 위치로 이동하는 모든 검증 사례에서 히스토리 영역은 목표 위치와 일치하는 콘텐츠를 100% 표시한다.
- **SC-003**: 히스토리 직접 스크롤과 미니맵 드래그 양쪽에서 현재 구간 표시의 위치가 전체 흐름 대비 실제 표시 위치와 5% 이내의 차이로 유지된다.
- **SC-004**: 500개의 프롬프트 및 출력 항목이 있는 run을 1440×900 Chromium/WebView 환경에서 검증할 때, 1회의 예열 후 5회 반복한 pointer 입력 시작부터 slider와 history 위치 반영 완료까지의 중앙값이 100ms 이하여야 한다.
- **SC-005**: 미니맵을 숨기고 다시 표시하는 모든 검증 사례에서 전체 흐름 대비 논리 viewport 시작 비율의 차이가 5% 이내이고, 콘텐츠나 조작 요소의 겹침이 발생하지 않는다.
- **SC-006**: 포인터와 키보드 각각의 탐색 테스트에서 사용자가 히스토리의 시작과 끝 및 임의의 중간 위치에 모두 도달할 수 있다.
- **SC-007**: 기존 agent-run 동작인 출력 스트리밍, 최신 출력 자동 추적, 프롬프트 입력 및 추가 패널 전환이 미니맵 표시 여부와 관계없이 회귀 없이 동작한다.

## Assumptions

- "세로로 2분할"은 세로 구분선을 기준으로 기존 히스토리를 왼쪽, 미니맵을 오른쪽에 두는 좌우 분할을 의미한다.
- 미니맵은 기본적으로 표시하여 기능을 발견할 수 있게 하며, 사용자가 패널을 사용하는 동안 토글 상태를 유지한다.
- 앱을 종료한 뒤에도 미니맵 표시 상태를 복원하는 영속 설정은 이번 기능 범위에 포함하지 않는다.
- 미니맵의 핵심 탐색 단위는 사용자 프롬프트와 agent의 사용자 대상 출력이다. 도구 호출, 권한 요청, 상태 메시지의 상세 재현은 기존 히스토리에서 확인한다.
- 미니맵은 기존 히스토리의 렌더링 결과를 축소 복제하는 화면이 아니라, 탐색에 필요한 발화 구조와 상대 위치를 간결하게 표현한다.
- 추가 agent 패널이 있는 경우 각 패널은 독립된 히스토리와 미니맵 상태를 가지며 현재 활성 패널만 화면에 표시된다.
- 기존 agent-run 히스토리의 스크롤 및 최신 출력 자동 추적 규칙은 유지한다.
