# Feature Specification: AW Git Commit 상세 한글 파일명 표시 수정

**Feature Branch**: `[015-fix-korean-filenames]`

**Created**: 2026-07-07

**Status**: Draft

**Input**: User description: "AW 내 git commit 디테일에서 한글 파일명이 정상 표시되지 않음. 한글이 `\\355\\202\\244\\354\\230\\244` 같은 백슬래시와 8진수 바이트 형식으로 표시됨."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 커밋 상세에서 한글 파일명 확인하기 (Priority: P1)

Agentic Workbench 사용자는 Git 커밋 상세 화면에서 변경된 파일 목록을 볼 때 한글 파일명과 경로를 원래 이름 그대로 읽고 싶다.

**Why this priority**: 커밋 상세의 핵심 목적은 변경 대상을 빠르게 이해하는 것이다. 한글 파일명이 `\\355\\202\\244\\354\\230\\244` 같은 백슬래시와 8진수 바이트 표기 또는 다른 이스케이프 형태로 보이면 사용자는 어떤 파일이 변경되었는지 즉시 판단할 수 없다.

**Independent Test**: 한글 파일명 또는 한글 경로가 포함된 커밋을 선택하고, 커밋 상세의 변경 파일 목록에서 해당 이름이 사람이 읽을 수 있는 한글로 표시되는지 확인하면 독립적으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 한글 파일명이 포함된 커밋이 존재함, **When** 사용자가 해당 커밋 상세를 열음, **Then** 변경 파일 목록에는 한글 파일명이 원래 문자로 표시되고 `\\355\\202\\244` 같은 8진수 바이트 표기가 노출되지 않는다.
2. **Given** 한글 디렉터리와 한글 파일명이 함께 포함된 경로가 존재함, **When** 사용자가 커밋 상세에서 변경 파일을 확인함, **Then** 전체 경로의 한글 부분이 읽을 수 있는 문자로 표시된다.
3. **Given** 한글과 영문, 숫자, 기호가 섞인 파일명이 존재함, **When** 사용자가 커밋 상세를 확인함, **Then** 파일명 구성 요소가 누락되거나 대체 문자로 바뀌지 않는다.

---

### User Story 2 - 파일명 표시 일관성 유지하기 (Priority: P2)

사용자는 커밋 상세 안의 파일 목록, 파일 헤더, diff 영역 등 같은 파일을 가리키는 위치에서 동일한 한글 파일명이 일관되게 표시되기를 원한다.

**Why this priority**: 한 화면 안에서 같은 파일명이 서로 다른 형태로 보이면 사용자는 diff와 파일 목록의 연결 관계를 신뢰하기 어렵다.

**Independent Test**: 한글 파일명이 포함된 커밋 상세에서 파일 목록 항목을 선택하고, 선택된 diff 또는 상세 영역의 파일명 표시가 같은 이름인지 비교하면 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 한글 파일명이 포함된 변경 파일이 있음, **When** 사용자가 커밋 상세에서 해당 파일을 선택함, **Then** 파일 목록과 상세 영역은 같은 한글 파일명을 표시한다.
2. **Given** 같은 커밋에 여러 한글 파일명이 포함됨, **When** 사용자가 파일 사이를 이동함, **Then** 각 파일의 표시명은 서로 뒤섞이지 않고 올바른 파일에 대응한다.
3. **Given** 한글 파일명이 긴 경로 안에 포함됨, **When** 화면이 좁은 상태에서 커밋 상세를 봄, **Then** 표시가 깨지지 않고 기존 UI의 줄임 또는 줄바꿈 규칙 안에서 읽을 수 있게 처리된다.

---

### User Story 3 - 한글 파일명 변경 이력 이해하기 (Priority: P3)

사용자는 파일 생성, 수정, 삭제뿐 아니라 이름 변경이 포함된 커밋에서도 이전 이름과 새 이름의 한글 표기를 이해하고 싶다.

**Why this priority**: 이름 변경 커밋은 두 경로를 비교해야 하므로 표시가 깨지면 변경 의미를 파악하기 어렵다. 다만 기본 파일 목록 표시가 먼저 정상화되어야 하므로 P3로 둔다.

**Independent Test**: 한글 파일명의 rename 또는 delete/create 이력이 포함된 커밋을 열고, 이전 경로와 새 경로가 읽을 수 있는 한글로 표시되는지 확인하면 독립적으로 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 한글 파일명이 다른 한글 파일명으로 변경된 커밋이 존재함, **When** 사용자가 커밋 상세를 열음, **Then** 이전 이름과 새 이름이 모두 읽을 수 있는 한글로 표시된다.
2. **Given** 영문 파일명이 한글 파일명으로 변경된 커밋이 존재함, **When** 사용자가 변경 파일 정보를 확인함, **Then** 변경 전후 이름이 각각 올바르게 구분되어 표시된다.

### Edge Cases

- 파일명에 한글, 공백, 괄호, 점, 하이픈, 밑줄, 숫자가 섞여 있어도 읽을 수 있는 표시를 유지해야 한다.
- 파일명이 `\\355\\202\\244\\354\\230\\244` 같은 백슬래시와 8진수 바이트 표기로 제공되는 경우 사용자가 읽을 수 있는 한글 표시명으로 보여야 한다.
- 파일명이 이미 정상적인 한글로 제공되는 경우 중복 변환으로 문자가 손상되면 안 된다.
- 한글이 아닌 파일명과 경로는 기존 표시와 의미가 달라지면 안 된다.
- 파일 경로가 매우 길거나 화면 폭이 좁아도 파일명 텍스트가 다른 UI 요소와 겹치지 않아야 한다.
- 커밋에 한글 파일명이 여러 개 포함되어도 각 항목은 고유한 파일을 정확히 가리켜야 한다.
- 한글 파일명을 포함한 삭제, 추가, 수정, rename 표시가 모두 커밋 상세에서 이해 가능해야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 AW의 Git 커밋 상세 화면에서 한글 파일명과 한글 경로를 사람이 읽을 수 있는 원래 문자로 표시해야 한다.
- **FR-002**: 시스템은 변경 파일 목록의 파일명 표시에서 `\\355\\202\\244\\354\\230\\244` 같은 백슬래시와 8진수 바이트 표기, 깨진 문자, 과도한 이스케이프 표기, 대체 문자로 인해 한글 파일명을 알아볼 수 없는 상태를 허용하지 않아야 한다.
- **FR-003**: 시스템은 파일 목록, 선택된 파일 표시, diff 영역의 파일 헤더처럼 같은 파일명을 보여주는 커밋 상세 내 위치에서 일관된 표시명을 사용해야 한다.
- **FR-004**: 시스템은 한글 파일명의 추가, 수정, 삭제 상태를 기존 변경 유형 정보와 함께 구분 가능하게 표시해야 한다.
- **FR-005**: 시스템은 한글 파일명 rename이 포함된 경우 이전 경로와 새 경로를 모두 읽을 수 있는 형태로 표시해야 한다.
- **FR-006**: 시스템은 한글이 포함되지 않은 기존 파일명 표시 경험을 변경하거나 악화시키면 안 된다.
- **FR-007**: 시스템은 이미 정상 표시 가능한 한글 파일명을 다시 처리해 문자를 손상시키면 안 된다.
- **FR-008**: 시스템은 한글과 ASCII 문자, 공백, 일반 기호가 혼합된 파일명에서도 전체 표시명을 보존해야 한다.
- **FR-009**: 시스템은 커밋 상세에서 한글 파일명이 긴 경로나 좁은 화면에 표시될 때도 텍스트가 다른 핵심 UI와 겹치지 않도록 해야 한다.
- **FR-010**: 시스템은 사용자가 한글 파일명이 포함된 커밋을 열 때 기존 커밋 상세 탐색 흐름을 유지해야 하며, 파일명 표시 문제로 커밋 상세 로딩이나 파일 선택이 실패하면 안 된다.

### Key Entities *(include if feature involves data)*

- **Commit Detail**: 사용자가 선택한 하나의 Git 커밋에 대한 상세 정보이며, 변경 파일 목록과 파일별 상세 정보를 포함한다.
- **Changed File Entry**: 커밋 안에서 추가, 수정, 삭제, 이름 변경된 파일 항목이다. 표시 가능한 경로, 변경 유형, 필요 시 이전 경로와 새 경로를 가진다.
- **Display File Path**: 사용자에게 보여지는 파일 경로 문자열이다. 원본 파일명을 사람이 읽을 수 있는 문자로 표현하며, 커밋 상세의 여러 위치에서 일관되게 사용된다.

## Constitution Alignment *(mandatory)*

- **Monorepo boundary**: 범위는 `apps/agentic-workbench`의 Git 커밋 상세 경험이다. 한글 경로 표시 규칙이 Git Explorer 또는 공통 Git UI에서도 재사용되어야 한다고 확인되면 순수 표시/정규화 규칙을 `packages/*` 또는 `crates/*`의 공통 경계로 분리한다.
- **Frontend layering**: 커밋 상세 화면은 `pages` 또는 해당 화면 조합에서 다루고, 파일 선택 및 표시 상호작용은 `features`, 커밋과 변경 파일 모델은 `entities` 경계에서 다룬다. 재사용 가능한 표시 primitive가 필요하면 `shared` 또는 기존 UI primitive를 사용한다.
- **Backend boundary**: Git 커밋 상세 데이터가 Tauri backend를 통해 제공되는 경우 경로 표시용 데이터 준비는 application service와 Git adapter 경계를 지켜야 한다. Tauri command가 Git 경로 처리나 표시 규칙을 직접 보유하면 안 된다.
- **Shared core vs UI**: 한글 파일명 표시 보정이 여러 앱에서 필요한 순수 규칙이면 headless core로 먼저 분리한다. AW 화면 배치, 선택 상태, diff 표시 UI는 AW local UI로 유지한다.
- **Persistence and safety**: 이 기능은 커밋 상세 표시를 개선하는 범위이며 파일명, Git 이력, 작업트리 파일을 변경하지 않는다. 경로는 현재 사용자가 열람 중인 repository/session 범위 안에서만 표시되어야 한다.
- **Documentation and Storybook**: 한글 파일명 상태가 포함된 커밋 상세 또는 파일 목록 UI 상태를 Storybook이나 동등한 UI 검증에 포함한다. 별도 문서가 필요하면 `docs/*.md`에 영어 파일명과 한국어 내용으로 작성한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 한글 파일명이 포함된 대표 커밋 10개를 확인했을 때 커밋 상세의 변경 파일 목록에서 한글 파일명이 100% 읽을 수 있는 문자로 표시되고, `\\355\\202\\244` 같은 8진수 바이트 표기는 0건이어야 한다.
- **SC-002**: 한글 경로가 포함된 파일의 목록 표시명과 상세/diff 표시명이 테스트된 모든 사례에서 동일한 파일을 가리키는 일관된 이름으로 표시된다.
- **SC-003**: 한글 파일명 추가, 수정, 삭제, rename 사례 각각에서 사용자는 5초 이내에 변경 대상 파일명을 식별할 수 있다.
- **SC-004**: 기존 영문/숫자 파일명만 포함된 커밋 상세 회귀 테스트에서 파일명 표시와 파일 선택 흐름의 실패가 0건이어야 한다.
- **SC-005**: 한글 파일명이 포함된 커밋 상세를 여는 과정에서 파일명 표시 문제로 인한 로딩 실패, 빈 파일 목록, 잘못된 파일 선택이 0건이어야 한다.

## Assumptions

- 문제는 AW의 Git 커밋 상세 화면에서 사용자가 보는 파일명 표시 품질에 한정하며, 실제 repository 파일명이나 Git 이력 변경은 범위에 포함하지 않는다.
- 한글 파일명은 UTF-8 기반의 일반적인 한국어 파일명과 경로를 우선 대상으로 하며, 현재 관찰된 주요 실패 형태는 백슬래시와 8진수 바이트로 표현된 경로 문자열이다.
- 커밋 상세 안에서 파일명이 나타나는 주요 위치는 변경 파일 목록, 선택된 파일 헤더, diff 관련 파일 표시 영역으로 본다.
- Git Explorer 등 다른 앱의 동일 문제가 확인되기 전까지 이번 명세의 기본 범위는 AW로 제한한다.
- 사용자 입력이나 파일명 변경 기능이 아니라 읽기 전용 커밋 상세 표시 개선으로 간주한다.
