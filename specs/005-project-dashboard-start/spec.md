# Feature Specification: 프로젝트 대시보드 시작화면

**Feature Branch**: `102-issue`

**Created**: 2026-07-02

**Status**: Draft

**Input**: User description: "github issue #102: 시작화면을 프로젝트 대시보드로 개선"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 최근 프로젝트로 즉시 복귀 (Priority: P1)

사용자는 앱을 열자마자 최근에 작업한 프로젝트 목록과 각 프로젝트의 주요 상태를 확인하고, 원하는 프로젝트 또는 최근 세션으로 바로 이동할 수 있다.

**Why this priority**: 시작화면의 핵심 가치는 사용자가 이전 작업을 빠르게 재개하는 것이다. 이 흐름만 완성되어도 기존 시작화면보다 즉시 체감되는 개선을 제공한다.

**Independent Test**: 최근 프로젝트와 세션 기록이 있는 상태에서 앱을 시작한 뒤, 대시보드에서 프로젝트 상태를 확인하고 최근 세션으로 진입할 수 있으면 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 최근에 사용한 프로젝트가 1개 이상 있는 사용자, **When** 앱 시작화면을 연다, **Then** 최근 프로젝트가 우선순위에 따라 표시되고 각 항목에서 프로젝트명, 최근 활동, 주요 작업 상태를 확인할 수 있다.
2. **Given** 최근 프로젝트에 이어서 진행 가능한 세션이 있는 사용자, **When** 해당 프로젝트의 재개 동작을 선택한다, **Then** 사용자는 해당 프로젝트의 최근 작업 맥락으로 이동한다.
3. **Given** 최근 프로젝트에 worktree 또는 변경 상태 요약이 있는 사용자, **When** 프로젝트 항목을 훑어본다, **Then** 사용자는 프로젝트에 들어가기 전에 진행 중인 작업의 존재 여부를 파악할 수 있다.

---

### User Story 2 - 주요 프로젝트 작업 시작 (Priority: P2)

사용자는 시작화면에서 새 프로젝트 생성, 기존 프로젝트 열기, 최근 세션 재개 같은 주요 action을 별도의 탐색 없이 실행할 수 있다.

**Why this priority**: 최근 작업이 없는 사용자나 새 작업을 시작하려는 사용자에게 시작화면이 막다른 곳이 되지 않도록 한다.

**Independent Test**: 최근 프로젝트 데이터와 무관하게 시작화면에서 주요 action을 선택하고 각 action의 목적지 또는 다음 단계가 명확히 표시되면 검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 앱 시작화면에 있다, **When** 새 프로젝트 생성을 선택한다, **Then** 새 프로젝트 생성 흐름으로 이동한다.
2. **Given** 사용자가 앱 시작화면에 있다, **When** 기존 프로젝트 열기를 선택한다, **Then** 프로젝트 선택 또는 열기 흐름으로 이동한다.
3. **Given** 최근 재개 가능한 세션이 있다, **When** 세션 재개 동작을 선택한다, **Then** 사용자는 해당 세션으로 이동한다.

---

### User Story 3 - 상태별 시작화면 이해 (Priority: P3)

사용자는 프로젝트가 없거나 데이터를 불러오는 중이거나 문제가 발생한 상태에서도 현재 상황과 다음에 할 수 있는 행동을 명확히 이해할 수 있다.

**Why this priority**: 시작화면은 앱 진입점이므로 빈 상태와 오류 상태가 불명확하면 신규 사용자와 복귀 사용자 모두 작업을 시작하지 못한다.

**Independent Test**: 프로젝트 없음, 로딩, 오류, 긴 목록 상태를 각각 표시했을 때 사용자가 다음 행동을 선택할 수 있고 화면이 깨지지 않으면 검증된다.

**Acceptance Scenarios**:

1. **Given** 등록된 프로젝트가 없는 사용자, **When** 앱 시작화면을 연다, **Then** 빈 상태가 표시되고 새 프로젝트 생성 또는 기존 프로젝트 열기 동작이 명확히 제공된다.
2. **Given** 프로젝트 정보를 불러오는 중인 사용자, **When** 시작화면을 본다, **Then** 진행 중 상태가 표시되고 기존 데이터가 아직 없다고 오해하지 않는다.
3. **Given** 프로젝트 정보를 불러오지 못한 사용자, **When** 시작화면을 본다, **Then** 오류 상태와 다시 시도 가능한 동작이 표시된다.

### Edge Cases

- 최근 프로젝트가 너무 많을 때 시작화면은 가장 관련성 높은 항목을 우선 표시하고 나머지는 기존 프로젝트 목록 또는 검색/탐색 흐름으로 이어져야 한다.
- 프로젝트 기록은 있으나 해당 프로젝트 위치를 더 이상 사용할 수 없을 때 사용자는 접근 불가 상태와 가능한 다음 행동을 확인할 수 있어야 한다.
- 프로젝트는 있으나 세션 기록이 없을 때 대시보드는 프로젝트 열기를 기본 행동으로 제공하고 세션 재개를 오해하게 만들지 않아야 한다.
- worktree, session, 변경 상태 중 일부 요약만 확인 가능한 경우에도 확인 가능한 정보와 확인 불가 정보를 구분해 표시해야 한다.
- 긴 프로젝트명, 긴 경로, 긴 세션 제목이 있는 경우에도 주요 action과 상태 정보가 서로 겹치거나 잘리지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 앱 시작 시 사용자를 프로젝트 중심 대시보드로 안내해야 한다.
- **FR-002**: 시스템은 최근 사용한 프로젝트를 사용자가 빠르게 훑어볼 수 있는 목록 또는 요약 영역으로 표시해야 한다.
- **FR-003**: 각 최근 프로젝트 항목은 프로젝트 식별 정보, 최근 활동 시점, 사용자가 다음 행동을 결정하는 데 필요한 주요 상태를 포함해야 한다.
- **FR-004**: 사용자는 대시보드에서 최근 프로젝트를 선택해 해당 프로젝트의 작업 화면 또는 프로젝트 상세 흐름으로 이동할 수 있어야 한다.
- **FR-005**: 사용자는 대시보드에서 재개 가능한 최근 세션이 있는 경우 해당 세션을 바로 재개할 수 있어야 한다.
- **FR-006**: 사용자는 대시보드에서 새 프로젝트 생성과 기존 프로젝트 열기를 명확한 주요 action으로 실행할 수 있어야 한다.
- **FR-007**: 시스템은 프로젝트별 worktree, session, 변경 상태 요약을 표시하되, 확인 가능한 정보만 신뢰 가능하게 보여주고 확인 불가 상태를 구분해야 한다.
- **FR-008**: 시스템은 최근 프로젝트가 없는 상태에서 빈 상태 메시지와 최소 1개 이상의 작업 시작 action을 제공해야 한다.
- **FR-009**: 시스템은 프로젝트 정보를 불러오는 동안 로딩 상태를 제공해야 하며, 로딩 상태가 빈 상태나 오류 상태로 오해되지 않아야 한다.
- **FR-010**: 시스템은 프로젝트 정보를 불러오지 못한 경우 오류 상태, 원인 이해에 필요한 짧은 설명, 다시 시도 또는 대체 action을 제공해야 한다.
- **FR-011**: 시작화면은 프로젝트 목록, 프로젝트 상세, worktree/session routing의 기존 역할과 중복되지 않도록 개요와 진입 중심의 정보를 제공해야 한다.
- **FR-012**: 시작화면은 반복 사용에 적합한 밀도 높은 operational UI여야 하며, 홍보성 landing page처럼 보이는 구성보다 실제 작업 진입을 우선해야 한다.
- **FR-013**: 대시보드의 주요 UI 상태는 검증 가능한 화면 예시 또는 상태별 검증 수단으로 확인 가능해야 한다.

### Key Entities

- **프로젝트**: 사용자가 작업을 수행하는 기본 단위. 이름, 위치 또는 식별자, 최근 활동, 접근 가능 여부, 주요 상태 요약을 가진다.
- **세션**: 프로젝트 안에서 이어서 진행할 수 있는 작업 맥락. 최근 활동, 재개 가능 여부, 연결된 프로젝트와의 관계를 가진다.
- **Worktree**: 프로젝트와 연결된 작업 공간 상태. 활성 여부, 변경 상태, 사용자가 프로젝트에 들어가기 전에 알아야 할 요약 정보를 제공한다.
- **대시보드 Action**: 새 프로젝트 생성, 기존 프로젝트 열기, 프로젝트 진입, 세션 재개, 다시 시도처럼 시작화면에서 직접 실행 가능한 사용자 행동이다.

## Constitution Alignment *(mandatory)*

- **Monorepo boundary**: 범위는 AW(`apps/agentic-workbench`) 시작화면 경험에 한정한다. 다른 앱으로 공유할 필요가 생기기 전까지 cross-app UI 공유는 의도적으로 피한다.
- **Frontend layering**: 시작화면의 screen-level UI는 `pages`, 프로젝트/세션 진입 같은 사용자 action은 `features`, 프로젝트와 세션 요약 모델은 `entities`, 재사용 가능한 표시 단위와 상태 UI는 `shared` 또는 기존 `components/ui` 경계를 따른다.
- **Backend boundary**: 프로젝트, 세션, worktree 상태 조회가 backend 변경을 요구할 경우 Tauri command는 inbound adapter로 유지하고, 조회 규칙은 application service에 위임하며, 파일/세션 상태 접근은 infrastructure adapter 뒤에 둔다.
- **Shared core vs UI**: 이 기능은 AW 전용 화면 개선이므로 shared UI 패키지 도입은 기본 범위가 아니다. 재사용이 필요한 경우 먼저 프로젝트/세션 요약 모델이나 순수 helper 수준에서 검토한다.
- **Persistence and safety**: 기존 프로젝트 기록, session, worktree 정보를 읽을 때는 프로젝트 root/path와 session owner 범위를 유지하며, 접근 불가한 항목은 실패로 중단하지 않고 상태로 표시한다.
- **Documentation and Storybook**: 대시보드 또는 재사용 가능한 주요 컴포넌트는 Storybook에 상태별 예시를 추가한다. 별도 문서가 필요하면 `docs/*.md`에 영어 파일명과 한국어 본문으로 작성한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 최근 프로젝트가 있는 사용자의 90% 이상이 앱 시작 후 10초 이내에 최근 프로젝트 또는 최근 세션으로 이동할 수 있다.
- **SC-002**: 신규 또는 프로젝트가 없는 사용자는 시작화면에서 15초 이내에 새 프로젝트 생성 또는 기존 프로젝트 열기 흐름을 찾을 수 있다.
- **SC-003**: 대표 UI 상태(최근 프로젝트 있음, 프로젝트 없음, 로딩, 오류, 긴 콘텐츠)가 모두 검증 예시로 확인 가능하다.
- **SC-004**: 사용자 평가 또는 리뷰에서 시작화면이 "작업 재개에 필요한 정보와 action을 제공한다"는 항목에 대해 80% 이상 긍정 응답을 얻는다.
- **SC-005**: 최근 프로젝트 항목을 훑어본 사용자의 80% 이상이 프로젝트에 들어가기 전에 세션 또는 변경 상태의 존재 여부를 올바르게 판단할 수 있다.

## Assumptions

- 시작화면은 AW 앱의 진입 화면을 의미하며 GE와 MA의 시작 경험은 이번 범위에 포함하지 않는다.
- 기존 프로젝트 목록, 프로젝트 상세, worktree/session routing은 유지하고, 대시보드는 개요와 빠른 진입 역할에 집중한다.
- 최근 프로젝트의 정렬은 최근 활동과 재개 가능성을 우선하는 방식이 적합하다고 가정한다.
- 프로젝트별 상태 요약은 모든 세부 정보를 표시하기보다 사용자가 다음 행동을 고르는 데 필요한 최소한의 신뢰 가능한 정보를 제공한다.
- 이 기능의 완료에는 상태별 UI 검증 또는 Storybook 예시가 포함된다.
