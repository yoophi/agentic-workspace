# 기능 명세: SDD 워크플로 단계 표시 및 제어

**Feature Branch**: `worktree-18c533ee42f34570`

**Created**: 2026-07-24

**Status**: Draft

**Input**: 사용자 설명: "SDD 작업을 원활하게 하기 위해서 현재의 작업 단계(specify, plan, task, implements)를 표시하고 SDD 작업을 할 수 있는 버튼들을 제공한다."

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 현재 SDD 진행 단계를 즉시 파악하기 (Priority: P1)

worktree session에서 SDD 작업을 하는 사용자는 활성 기능의 산출물 상태를 보고 현재 단계와 다음에 진행할 수 있는 단계를 한눈에 파악한다. 사용자는 파일 목록을 직접 비교하거나 명령 순서를 기억하지 않아도 된다.

**우선순위 이유**: 올바른 다음 단계를 알 수 있어야 SDD 작업을 중단 없이 이어 갈 수 있다.

**독립 테스트**: spec만 있는 활성 기능, plan까지 있는 활성 기능, tasks까지 있는 활성 기능을 각각 열어 단계 표시와 다음 단계 안내가 정확한지 확인한다.

**인수 시나리오**:

1. **Given** 활성 기능에 spec 산출물만 있는 상태, **When** 사용자가 SDD 제어 영역을 열면, **Then** specify는 완료로, plan은 다음 단계로, tasks와 implement는 대기 단계로 표시된다.
2. **Given** 활성 기능에 spec, plan, tasks 산출물이 있는 상태, **When** 사용자가 SDD 제어 영역을 열면, **Then** implement가 다음 단계로 표시되고 이전 단계의 완료 상태를 확인할 수 있다.
3. **Given** Speckit 패널에 여러 기능이 표시된 상태, **When** 현재 활성 기능이 식별되면, **Then** 해당 기능 항목은 다른 기능과 구분되는 하이라이트와 활성 상태 표기로 표시된다.
4. **Given** 활성 기능을 식별할 수 없거나 SDD 산출물이 없는 상태, **When** 사용자가 SDD 제어 영역을 열면, **Then** 현재 상태를 설명하는 빈 상태와 시작 방법이 표시되고 화면이 깨지지 않는다.
5. **Given** 활성 기능 추적 정보가 없거나 유효하지 않은 상태, **When** 사용자가 SDD 작업을 시작하려고 하면, **Then** 시스템은 활성 기능 설정에 필요한 초기 프롬프트를 제안하거나 주입하여 사용자가 올바른 시작 흐름을 이어 갈 수 있게 한다.

---

### 사용자 스토리 2 - 다음 SDD 작업을 버튼으로 시작하기 (Priority: P1)

사용자는 현재 단계에 맞는 제어 버튼을 선택하여 specify, plan, tasks, implement 작업을 시작할 수 있다. 시작된 작업은 현재 worktree와 활성 기능의 맥락을 유지하며 기존 에이전트 작업 흐름에서 확인할 수 있어야 한다.

**우선순위 이유**: 단계 표시만으로는 반복되는 SDD 명령 입력과 맥락 전환을 줄일 수 없다.

**독립 테스트**: 각 단계가 가능한 대표 상태에서 해당 버튼을 선택하고, 올바른 SDD 작업 요청이 현재 session의 에이전트 작업 흐름에 전달되는지 확인한다.

**인수 시나리오**:

1. **Given** SDD 시작 준비가 된 worktree session, **When** 사용자가 Specify 버튼을 선택하면, **Then** 기능 설명을 제공하거나 작성할 수 있는 시작 흐름이 열리고 그 내용으로 specify 작업이 요청된다.
2. **Given** spec이 완료되고 사용자 승인을 받은 상태, **When** 사용자가 Plan 버튼을 선택하면, **Then** 현재 활성 기능을 대상으로 plan 작업이 요청된다.
3. **Given** plan이 완료되고 사용자 승인을 받은 상태, **When** 사용자가 Tasks 버튼을 선택하면, **Then** 현재 활성 기능을 대상으로 tasks 작업이 요청된다.
4. **Given** tasks가 완료되고 사용자 승인을 받은 상태, **When** 사용자가 Implement 버튼을 선택하면, **Then** 현재 활성 기능의 미완료 작업을 대상으로 implement 작업이 요청된다.

### 사용자 스토리 3 - 단계 순서와 승인 게이트를 안전하게 지키기 (Priority: P2)

사용자는 아직 준비되지 않은 단계나 검토 승인이 필요한 다음 단계를 실수로 실행하지 않는다. 이미 완료된 단계를 다시 실행하려 할 때는 영향과 선택지를 이해할 수 있다.

**우선순위 이유**: SDD 산출물을 덮어쓰거나 검토 단계를 건너뛰면 작업 품질과 신뢰성이 저하된다.

**독립 테스트**: spec이 없는 상태에서 Plan을 선택하려는 경우, 검토 승인이 없는 상태에서 다음 단계를 선택하는 경우, 완료된 단계를 다시 실행하는 경우를 각각 확인한다.

**인수 시나리오**:

1. **Given** 선행 산출물이 없는 상태, **When** 사용자가 이후 단계를 확인하면, **Then** 해당 버튼은 실행할 수 없고 필요한 선행 조건이 설명된다.
2. **Given** spec 또는 plan 검토가 필요한 상태, **When** 사용자가 다음 생성 단계를 선택하면, **Then** 시스템은 사용자의 승인 또는 명시적 진행 의사를 확인한 뒤에만 요청을 전달한다.
3. **Given** 이미 완료된 단계, **When** 사용자가 해당 단계를 다시 실행하려고 하면, **Then** 시스템은 기존 산출물에 영향을 줄 수 있음을 알리고 취소하거나 다시 실행할 수 있게 한다.

### 엣지 케이스

- 기능 폴더에는 일부 산출물만 있고, 파일 이름 또는 내용이 읽히지 않을 수 있다.
- 활성 기능 표시와 화면에 선택된 기능이 서로 다를 수 있다.
- 활성 기능이 현재 Speckit 패널의 필터 또는 정렬 결과에 없을 수 있다.
- 에이전트 실행 중, 실행 대기 중, 또는 실행을 시작할 수 없는 session일 수 있다.
- 사용자가 다른 창이나 에이전트에서 산출물을 생성·삭제하여 표시 상태가 오래되었을 수 있다.
- `.specify/feature.json`이 없거나, 잘못된 형식이거나, 존재하지 않는 기능 경로를 가리킬 수 있다.
- 작업 요청 전달이 실패하거나 사용자가 시작 흐름을 취소할 수 있다.
- tasks 산출물은 있으나 완료되지 않은 작업이 없거나, 체크 가능한 작업 항목이 없을 수 있다.

### 사용자 스토리 4 - tasks 작업을 칸반으로 분리해 보기 (Priority: P3)

사용자는 활성 기능의 `tasks.md`를 열 때 체크박스 작업을 칸반 형식으로 확인하고, 아직 진행할 작업 또는 완료된 작업만 골라 볼 수 있다. 사용자는 긴 markdown 목록을 직접 훑지 않고 다음 작업과 완료 이력을 구분한다.

**우선순위 이유**: SDD 단계 제어 이후에는 구현 대상과 완료된 작업을 빠르게 구분하는 것이 다음 행동을 정하는 데 중요하다.

**독립 테스트**: 완료·미완료 체크박스가 섞인 `tasks.md`를 선택한 뒤 칸반 모드에서 두 열과 필터를 확인하고, 각각의 필터가 해당 상태의 작업만 표시하는지 확인한다.

**인수 시나리오**:

1. **Given** 체크박스 작업이 있는 `tasks.md`가 선택된 상태, **When** 사용자가 칸반 모드로 전환하면, **Then** 미완료 작업과 완료 작업이 구분된 열로 표시된다.
2. **Given** 칸반 모드가 열린 상태, **When** 사용자가 진행할 작업만 보기로 전환하면, **Then** 미완료 작업만 표시된다.
3. **Given** 칸반 모드가 열린 상태, **When** 사용자가 완료된 작업만 보기로 전환하면, **Then** 완료 작업만 표시된다.
4. **Given** 체크박스 작업이 없거나 `tasks.md`가 아닌 문서가 선택된 상태, **When** 사용자가 칸반 모드를 확인하면, **Then** 작업이 없음을 설명하는 안전한 상태가 표시되고 markdown 미리보기는 계속 사용할 수 있다.
5. **Given** `tasks.md`에 완료된 작업과 미완료 작업이 서로 다른 섹션에 있는 상태, **When** 사용자가 작업 필요 보기로 전환하면, **Then** 미완료 작업을 가진 섹션의 헤더와 관련 설명만 표시되고 완료된 작업·완료 작업만 가진 섹션·그 밖의 문서는 표시되지 않는다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 현재 worktree의 활성 SDD 기능을 식별할 때 `.specify/feature.json`을 최우선 출처로 사용해야 한다.
- **FR-001a**: 시스템은 Speckit 패널에 표시된 기능 중 현재 활성 SDD 기능을 다른 기능과 시각적으로 구분되는 하이라이트와 접근 가능한 상태 표기로 표시해야 한다.
- **FR-001b**: 시스템은 `.specify/feature.json`이 없거나 유효하지 않거나 존재하지 않는 기능을 가리킬 때 이를 명확히 표시하고, 자동으로 다른 기능을 활성 기능으로 간주하지 않아야 한다.
- **FR-002**: 시스템은 활성 기능의 specify, plan, tasks, implement 단계를 순서대로 표시하고 각 단계를 완료, 현재/다음, 대기, 실행 불가 중 구분 가능한 상태로 보여야 한다.
- **FR-003**: 시스템은 존재하는 spec, plan, tasks 산출물과 tasks 진행 상태를 기반으로 사용자가 다음에 수행할 수 있는 단계를 안내해야 한다.
- **FR-004**: 시스템은 사용자가 SDD 제어 영역에서 Specify, Plan, Tasks, Implement 작업을 시작할 수 있는 버튼 또는 동등한 명확한 제어를 제공해야 한다.
- **FR-005**: Specify 작업을 시작할 때 시스템은 사용자가 기능 설명을 입력하거나 확인할 기회를 제공한 뒤 해당 설명으로 작업 요청을 전달해야 한다.
- **FR-006**: Plan, Tasks, Implement 작업 요청은 현재 표시된 활성 기능과 현재 worktree session의 맥락을 사용해야 한다.
- **FR-007**: 시스템은 선행 단계가 완료되지 않은 작업의 시작 제어를 실행 불가로 표시하고 필요한 선행 조건을 설명해야 한다.
- **FR-008**: 시스템은 spec 검토 후 plan 진행, plan 검토 후 tasks 진행에 필요한 사용자 승인 게이트를 명시적으로 표시하고, 승인 또는 명시적 진행 의사 없이 다음 단계 요청을 전달하지 않아야 한다.
- **FR-009**: 시스템은 완료된 단계를 다시 시작하려는 경우 기존 산출물에 영향을 줄 수 있음을 알리고 사용자의 확인을 받아야 한다.
- **FR-010**: 시스템은 작업 요청이 접수되었는지, 실행 중인지, 실패했는지, 취소되었는지를 사용자가 확인할 수 있게 표시해야 한다.
- **FR-011**: 시스템은 활성 기능이 없거나 산출물이 없거나 일부 파일을 읽지 못해도 안전한 빈 상태 또는 오류 상태를 표시하고 다른 worktree session 기능을 방해하지 않아야 한다.
- **FR-012**: 시스템은 외부 변경 후 새로고침 또는 자동 갱신 시 단계 표시와 실행 가능 제어를 최신 산출물 상태에 맞춰 갱신해야 한다.
- **FR-013**: 기능은 단계 판별, 활성 기능 선택, 각 단계 요청, 승인·재실행 확인, 빈·오류·갱신 상태에 대한 검증을 포함해야 한다.
- **FR-014**: 시스템은 활성 기능이 설정되지 않아 SDD 작업을 시작할 수 있을 때, 사용자가 활성 기능을 설정하거나 specify 작업을 시작하는 데 필요한 초기 프롬프트를 제안하거나 현재 session의 입력 영역에 주입할 수 있어야 한다.
- **FR-015**: 시스템은 초기 프롬프트를 자동으로 실행하지 않고, 사용자가 내용을 확인·편집·취소하거나 명시적으로 전송할 수 있게 해야 한다.
- **FR-016**: 시스템은 선택된 `tasks.md`의 체크박스 작업을 완료 여부에 따라 미완료와 완료 칸반 열로 표시할 수 있어야 한다.
- **FR-017**: 시스템은 칸반에서 미완료 작업만 보기와 완료 작업만 보기 모드를 제공해야 한다.
- **FR-018**: 시스템은 체크박스가 없는 `tasks.md` 또는 `tasks.md`가 아닌 문서에 대해 명확한 빈 상태를 표시하고 기존 markdown 미리보기를 방해하지 않아야 한다.
- **FR-019**: 칸반의 작업 항목은 원본 `tasks.md`에서 읽은 상태를 표시하는 읽기 전용 보기이며, 칸반에서 체크 상태를 변경하지 않아야 한다.
- **FR-020**: 시스템은 `tasks.md`에 대해 작업 필요 보기 모드를 제공해야 하며, 이 모드에서는 미완료 체크박스 작업이 있는 markdown 영역만 표시해야 한다.
- **FR-021**: 작업 필요 보기에서 시스템은 각 미완료 작업과 같은 섹션에 속한 헤더 및 작업 이해에 필요한 비작업 문맥만 표시해야 한다.
- **FR-022**: 작업 필요 보기에서 시스템은 완료된 체크박스 작업, 완료된 작업만 포함한 섹션의 헤더·설명, 미완료 작업과 관련 없는 기타 markdown 내용을 표시하지 않아야 한다.

### 핵심 엔티티

- **SDD 활성 기능**: 현재 worktree에서 SDD 제어 대상으로 선택되거나 추적되는 기능 산출물 묶음이다.
- **활성 기능 추적 정보**: `.specify/feature.json`에 기록된 활성 기능 경로이며, SDD 제어 대상 판별의 최우선 출처다.
- **SDD 단계**: specify, plan, tasks, implement 중 하나이며, 선행 단계 및 검토 승인 조건을 가진다.
- **단계 상태**: 각 SDD 단계의 완료 여부, 현재 또는 다음 단계 여부, 실행 가능 여부와 사유를 나타낸다.
- **단계 작업 요청**: 사용자가 버튼으로 시작한 SDD 작업의 종류, 대상 기능, worktree session 맥락, 요청 상태를 나타낸다.
- **승인 게이트**: spec 또는 plan 산출물을 검토한 사용자가 다음 단계로 진행하는 것을 명시적으로 확인하는 결정이다.
- **초기 SDD 프롬프트**: 활성 기능이 없거나 유효하지 않을 때, 활성 기능 설정 또는 specify 작업 시작에 필요한 맥락을 사용자에게 제공하는 편집 가능한 제안이다.
- **칸반 작업 항목**: `tasks.md`의 markdown 체크박스에서 추출한 읽기 전용 작업이며, 텍스트, 완료 여부, 원본 순서를 가진다.
- **작업 필요 영역**: 하나 이상의 미완료 체크박스 작업과 그 작업의 상위 헤더·연속 문맥으로 구성된 읽기 전용 markdown 부분집합이다.

## Constitution Alignment *(필수)*

- **Monorepo boundary**: 범위는 `apps/agentic-workbench`와 `specs/031-sdd-workflow-controls` 산출물에 한정한다. 다른 앱 또는 공유 패키지 변경은 포함하지 않는다.
- **Frontend layering**: worktree 및 SDD 산출물 상태 모델은 `entities` 또는 기존 feature 모델에, 단계 제어와 요청 상태는 `features`, session 화면 조합은 `pages` 또는 기존 worktree workspace 화면에 둔다. 재사용 가능한 UI만 `shared` 또는 `components/ui` 경계를 따른다.
- **Backend boundary**: 새 session·파일·명령 실행 연동이 필요할 경우, 순수 단계 판별 규칙은 domain/application 경계에 두고 Tauri 명령은 입력 검증과 위임만 담당한다. 파일·session 접근은 infrastructure/port 경계를 따른다.
- **Shared core vs UI**: AW 전용 작업 흐름이므로 새 cross-app UI는 만들지 않는다. 단계 판별 규칙이 다른 화면에서도 필요해질 경우에만 순수 모델을 우선 재사용한다.
- **Persistence and safety**: 대상 기능과 worktree session의 소유·경로 범위를 확인해야 한다. 승인 상태나 실행 요청을 저장한다면 기존 저장소 경계와 session 범위 검증을 사용한다.
- **Documentation and Storybook**: 새로 재사용 가능한 제어 컴포넌트를 만들면 원자적 Storybook 예제를 추가한다. 화면 전용 UI는 focused test와 worktree session 수동 검증을 포함한다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: spec만 존재, plan까지 존재, tasks까지 존재한 대표 활성 기능 3개에서 사용자가 현재/다음 SDD 단계를 5초 안에 식별할 수 있다.
- **SC-001a**: 여러 기능이 표시된 대표 Speckit 패널에서 사용자는 현재 활성 기능을 2초 안에 하이라이트와 상태 표기로 식별할 수 있다.
- **SC-001b**: `.specify/feature.json`이 없거나 유효하지 않은 대표 시나리오 100%에서 시스템은 잘못된 활성 기능을 표시하지 않고 사용자가 편집 가능한 초기 SDD 프롬프트를 확인할 수 있다.
- **SC-007**: 완료·미완료 체크박스가 섞인 대표 `tasks.md`에서 칸반은 원본 체크박스 작업 100%를 올바른 상태 열에 표시한다.
- **SC-008**: 진행할 작업만 보기와 완료된 작업만 보기에서 선택하지 않은 상태의 작업은 0건 표시된다.
- **SC-009**: 완료 작업만 있는 섹션, 완료된 체크박스 행, 미완료 작업과 무관한 markdown 영역은 작업 필요 보기에서 0건 표시된다.
- **SC-010**: 미완료 작업이 있는 각 섹션은 작업 필요 보기에서 해당 작업을 식별할 수 있는 헤더와 함께 100% 표시된다.
- **SC-002**: 각 단계가 실행 가능한 대표 상태에서 사용자가 해당 작업 요청을 2회 이하의 명시적 동작으로 시작할 수 있다.
- **SC-003**: 선행 산출물 또는 승인 게이트가 없는 대표 시나리오 100%에서 이후 단계 작업 요청이 전달되지 않는다.
- **SC-004**: 활성 기능 없음, 산출물 일부 누락, 파일 읽기 실패, 외부 변경 후 갱신의 대표 시나리오 100%에서 화면이 계속 표시되고 사용자가 상태 또는 재시도 방법을 확인할 수 있다.
- **SC-005**: 단계 요청 성공·진행·실패·취소의 대표 상태마다 사용자는 요청 후 2초 이내에 현재 결과를 확인할 수 있다.
- **SC-006**: focused 자동 테스트와 문서화된 수동 검증에서 specify, plan, tasks, implement의 요청 대상이 현재 worktree와 선택된 활성 기능에 일치함을 확인할 수 있다.

## 가정

- 첫 버전은 하나의 worktree session에서 한 번에 하나의 활성 SDD 기능을 제어 대상으로 삼는다.
- 활성 기능은 `.specify/feature.json`만을 최우선 출처로 사용한다. 파일이 없거나 유효하지 않은 경우에는 선택된 Speckit 기능으로 자동 대체하지 않고, 초기 SDD 프롬프트를 통해 사용자가 다음 작업을 결정한다.
- 사용자 승인 게이트는 작업 요청 전에 표시되는 명시적 확인으로 충분하며, 별도의 조직 승인 시스템 연동은 범위 밖이다.
- 기존 에이전트 실행·프롬프트 전달 흐름은 SDD 단계 요청을 표시하고 실행하는 기본 경로로 계속 사용한다.
- 자동으로 SDD 작업을 연속 실행하거나 승인 없이 다음 단계로 넘어가는 기능은 범위 밖이다.
- 초기 SDD 프롬프트는 자동 전송하지 않으며, 사용자가 명시적으로 전송할 때만 기존 에이전트 실행 흐름으로 전달한다.
- 칸반과 작업 필요 보기는 현재 선택한 `tasks.md`의 읽기 전용 보조 보기이며, 기존 Markdown 보기와 별개의 선택 가능한 모드다. 기본값은 Markdown 보기이며 task 파일을 수정하지 않는다.
- 작업 필요 보기는 원본 markdown을 수정하지 않고, 미완료 체크박스의 상위 heading 경계를 기준으로 필요한 영역을 계산한다. heading이 없는 작업은 해당 작업 자체와 바로 인접한 설명만 표시한다.
